# ADR-0008: Use cargo-dist for binary distribution

## Status
Accepted

## Context
awrust-cli needs a distribution mechanism that delivers pre-built binaries to end users across macOS (Intel + Apple Silicon), Linux (x86_64 + aarch64), and Windows. Users should not need the Rust toolchain installed.

We want:
- Cross-platform binary releases triggered by git tags
- Installer scripts for frictionless one-liner installs
- Homebrew tap support for macOS/Linux users
- Minimal ongoing maintenance burden
- GitHub Releases as the artifact host

## Decision
Use **cargo-dist** (v0.31.0) to automate cross-platform binary builds, installer generation, and GitHub Releases. On every semver git tag push, a Release workflow builds platform archives, generates shell/powershell installer scripts, publishes a Homebrew formula to `awrust/homebrew-tap`, and creates a GitHub Release with all artifacts.

Target matrix:
- `aarch64-apple-darwin`
- `x86_64-apple-darwin`
- `aarch64-unknown-linux-gnu`
- `x86_64-unknown-linux-gnu`
- `x86_64-pc-windows-msvc`

Installers:
- `shell` — `curl | sh` one-liner for macOS/Linux
- `powershell` — `irm | iex` one-liner for Windows
- `homebrew` — formula published to `awrust/homebrew-tap`

## Alternatives considered

- **Manual GitHub Actions + release workflow**: Full control over the CI pipeline, but requires writing and maintaining cross-compilation matrix YAML, archive scripts, checksum generation, and installer templates by hand. High initial effort and ongoing maintenance for each new target or installer format.
- **Homebrew tap only (manual formula)**: Familiar to macOS users, but requires maintaining a separate repository with SHA-pinned formulas updated on every release. Does not cover Windows or curl-based installs. cargo-dist generates the Homebrew formula automatically, so a standalone tap adds effort without benefit.
- **`cargo install` via crates.io**: Zero distribution infrastructure needed. However, requires users to have the Rust toolchain installed, compile from source on every install/update, and wait for compilation. Not viable as the primary distribution channel for non-Rust users.
- **Cross-rs + custom CI matrix**: Cross-compilation via Docker containers. More flexible for exotic targets (e.g., musl, ARM variants), but heavier CI setup, slower builds, and does not generate installers or release artifacts — still needs a separate release pipeline on top.
- **Nix flake**: Reproducible builds and atomic installs for Nix users. Small audience overlap with the target user base (developers using AWS-compatible tooling). Can be added later as a supplementary channel without conflicting with cargo-dist.

## Consequences
Positive:
- Single `dist generate` command produces the entire release workflow — no hand-written CI
- Tag-driven releases: `git tag v0.2.0 && git push --tags` triggers build, package, and publish
- Shell and PowerShell installers give users a one-liner install without Rust
- Homebrew formula auto-published to `awrust/homebrew-tap` on each release
- `[profile.dist]` with `lto = "thin"` produces optimized binaries
- Adding new targets or installers is a config change in `dist-workspace.toml`, not a CI rewrite

Negative:
- Dependency on the cargo-dist project for CI workflow generation and installer templates
- Release workflow is auto-generated — custom CI steps require post-generation patching or cargo-dist's extension points
- Homebrew tap requires a separate `awrust/homebrew-tap` repository with write access from CI
