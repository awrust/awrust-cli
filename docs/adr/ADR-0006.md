# ADR-0006: Use thiserror for error type derivation

## Status
Accepted

## Context
awrust-cli has a unified `Error` enum covering four failure modes: HTTP transport errors, I/O errors, XML parsing errors, and API errors (non-2xx responses). Each variant needs `Display` and `Error` trait implementations, and several need `From` conversions for the `?` operator.

We want:
- Minimal boilerplate for error type definitions
- Automatic `From` implementations for error propagation
- Clear, formatted error messages for stderr output

## Decision
Use **thiserror** for deriving `Error` and `Display` on the unified error type.

## Alternatives considered
- **Manual implementations**: Verbose. Four variants with Display, Error, and From impls is ~40 lines of boilerplate.
- **anyhow**: Designed for applications, not libraries. Erases error types, making it harder to match on specific error kinds if needed later.
- **eyre**: Similar to anyhow. Good for error reports but overkill for a CLI with simple error paths.
- **snafu**: More powerful but heavier API. Unnecessary complexity for a flat error enum.

## Consequences
Positive:
- Error enum definition is self-documenting: `#[error("{0}")]` shows the format inline
- `#[from]` generates `From` impls, enabling clean `?` propagation
- Zero runtime cost — all code generation happens at compile time

Negative:
- Proc macro dependency adds to compile time
- Less flexibility than manual implementations for complex error formatting
