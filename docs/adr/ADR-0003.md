# ADR-0003: Use Reqwest as the HTTP client

## Status
Accepted

## Context
awrust-cli communicates with the awrust server entirely over HTTP. It needs to send GET, PUT, and DELETE requests with optional bodies (XML for bucket creation, file bytes for uploads) and read responses (XML for listings, bytes for downloads).

We want:
- Simple API for common HTTP methods
- Async support (compatible with Tokio)
- Ability to send and receive both text and binary bodies

## Decision
Use **Reqwest** as the HTTP client.

## Alternatives considered
- **hyper**: Lower-level; requires manually building requests, handling connections, and managing body streams. Too much boilerplate for a CLI tool.
- **ureq**: Blocking-only; incompatible with our async architecture (ADR-0001).
- **surf**: Less maintained; smaller ecosystem.
- **isahc**: Capable but less widely used; curl-based backend adds native dependency complexity.

## Consequences
Positive:
- Clean, builder-pattern API for HTTP requests
- Built on Tokio + Hyper — proven async HTTP stack
- Handles connection pooling, redirects, and TLS transparently

Negative:
- Large dependency tree (Hyper, TLS, encoding)
- Pulls in TLS support we don't need (awrust is localhost/plaintext), but the compile cost is acceptable
