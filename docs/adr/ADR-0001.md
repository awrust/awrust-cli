# ADR-0001: Use Tokio as the async runtime

## Status
Accepted

## Context
awrust-cli makes HTTP requests to a remote awrust server. The Rust HTTP client ecosystem (reqwest, hyper) is async-first. We need a runtime that enables non-blocking I/O for HTTP requests and file operations (upload/download).

We want:
- Compatibility with the async HTTP client stack
- Async filesystem I/O for file uploads and downloads
- A runtime that is already proven in the awrust ecosystem

## Decision
Use **Tokio** as the async runtime for awrust-cli.

## Alternatives considered
- **Blocking reqwest**: Simpler initially, but prevents async file I/O and limits future concurrency (e.g., parallel uploads).
- **async-std**: Smaller ecosystem adoption; reqwest's default runtime is Tokio.
- **smol**: Lightweight but less ecosystem support for HTTP clients.

## Consequences
Positive:
- Direct compatibility with reqwest's default async transport
- `tokio::fs` provides async file reads/writes for `cp` operations
- Consistent with the awrust server's runtime choice

Negative:
- Foundational dependency; adds compile time
- Requires `#[tokio::main]` entry point and async throughout
