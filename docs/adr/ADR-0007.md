# ADR-0007: Use Behave for BDD integration testing

## Status
Accepted

## Context
awrust-cli needs integration tests that verify the full CLI workflow: argument parsing, HTTP communication with awrust, XML handling, file I/O, and output formatting. Tests should run against a real awrust instance, not mocks, to catch real integration issues.

We want:
- Human-readable test scenarios that serve as a living spec
- End-to-end testing of the CLI binary as a subprocess
- Real awrust Docker container as the test backend
- Consistency with the testing approach used in awrust-s3

## Decision
Use **Behave** (Python BDD framework) for integration testing. Tests invoke the `awr` binary via subprocess against a real `ghcr.io/awrust/awrust` Docker container managed by `environment.py`.

## Alternatives considered
- **Rust integration tests (assert_cmd + wiremock)**: Tests against mocks, not the real service. Misses protocol-level bugs and XML format issues that only surface against the real server.
- **Rust integration tests (assert_cmd + testcontainers)**: Viable but adds heavyweight Rust test dependencies (testcontainers, bollard, Docker API bindings). Python subprocess calls are simpler for CLI testing.
- **Shell scripts**: Fragile, no structured assertions, hard to maintain.
- **pytest**: Capable but lacks the Gherkin spec layer. Behave's feature files are self-documenting.

## Consequences
Positive:
- Feature files read as a specification: Given/When/Then scenarios document CLI behavior
- Tests hit the real awrust service — catches actual compatibility issues
- Consistent with awrust-s3 testing conventions (same framework, similar patterns)
- Lightweight: only `behave` pip package needed (no boto3 since we test the CLI, not the API)

Negative:
- Requires Python and Docker in the development and CI environment
- Subprocess-based testing is slower than in-process Rust tests
- Step definitions are in Python, separate from the Rust codebase
